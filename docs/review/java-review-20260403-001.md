## 二、问题发现

### 🔴 严重问题

#### 1. 线程内存泄漏风险 - `TracingManager.java:11`

**位置**: `jwsch-srv/src/main/java/cn/itcraft/jwsch/srv/tracing/TracingManager.java:11`

**问题**: `ThreadLocal<TracingContext>` 未清理，存在内存泄漏风险。

```java
private static final ThreadLocal<TracingContext> CURRENT_CONTEXT = new ThreadLocal<>();
```

`endTrace()` 方法虽然调用了 `remove()`，但在异常场景下可能不会执行。

**修复建议**:
```java
try {
    // trace logic
} finally {
    CURRENT_CONTEXT.remove();
}
```
或使用 `try-with-resources` 模式。

---

#### 2. 线程池未命名 - `ResponseMapping.java:35`

**位置**: `jwsch-srv/src/main/java/cn/itcraft/jwsch/srv/router/ResponseMapping.java:35`

```java
this.scheduler = Executors.newSingleThreadScheduledExecutor(r -> {
    Thread t = new Thread(r, "response-mapping-timeout");
    t.setDaemon(true);
    return t;
});
```

虽然命名了，但 daemon 线程在 JVM 关闭时可能被强制终止，导致超时任务无法完成。

**修复建议**: 增加 shutdown hook 确保任务完成。

---

#### 3. 定时任务未处理异常 - `DefaultTopicStatsManager.java:49`

**位置**: `jwsch-srv/src/main/java/cn/itcraft/jwsch/srv/stats/DefaultTopicStatsManager.java:49`

```java
scheduler.scheduleAtFixedRate(
    this::checkTopicLimit,  // 若抛出异常，后续任务将停止执行
    cleanupIntervalSeconds,
    cleanupIntervalSeconds,
    TimeUnit.SECONDS
);
```

**修复建议**: 在 `checkTopicLimit()` 内部捕获所有异常。

---

### 🟡 中等问题

#### 4. 缓存未设置上限 - `GuavaCache.java:20`

**位置**: `jwsch-common/src/main/java/cn/itcraft/jwsch/common/cache/GuavaCache.java:20`

```java
if (config.getMaximumSize() < Long.MAX_VALUE) {
    builder.maximumSize(config.getMaximumSize());
}
```

当 `config.getMaximumSize()` 为 `Long.MAX_VALUE` 时，缓存无上限，可能导致 OOM。

**修复建议**: 设置合理的默认上限。

---

#### 5. 重复代码 - `LoadBalance` 实现

**位置**: `RoundRobinLoadBalance.java`, `ConsistentHashLoadBalance.java`

两个类都有 `reset()` 和 `resetAll()` 方法，代码重复。

**修复建议**: 在抽象基类中抽取公共逻辑。

---

#### 6. `SimpleDateFormat` 线程安全问题

**未发现直接使用**，但在配置解析中需警惕。已确认使用 `DateTimeFormatter` 或未使用日期格式化。

---

#### 7. 日志缺少换行符校验 - 多处

**位置**: 多处日志打印

```java
LOGGER.info("WebSocket connected: connectionId={}, remote={}", 
    connectionId, ctx.channel().remoteAddress());
```

未校验 `connectionId` 和 `remoteAddress` 是否包含 `\r\n`，存在日志注入风险。

**修复建议**: 增加日志脱敏工具类。

---

### 🟢 轻微问题

## 四、线程安全评估

### 已正确实现

1. `ConcurrentHashMap` 广泛使用 ✅
2. `CopyOnWriteArrayList` 用于监听器列表 ✅
3. `AtomicInteger` / `AtomicLong` 用于计数器 ✅
4. `volatile` 标志位 (`started`, `shutdown`) ✅
5. 引用计数管理 (`SharedEventLoopManager.refCount`) ✅

### 潜在风险

1. `RoundRobinLoadBalance.counterMap` - `AtomicInteger` 可能溢出（极低概率）
2. `IdGenerator.COUNTER` - 静态 `AtomicLong` 重启后重置

## 五、性能优化建议

### 已优化的点

1. `PacketWriter.write()` 零拷贝 ✅
2. `broadcastToTopic()` 先编码一次再分发 ✅
3. `TopicSubscription` 反向索引加速查询 ✅

### 建议优化

1. `StringUtils` 方法较少，可考虑引入 Apache Commons Lang
2. `PacketHeader.getTopicBytes()` 缓存机制很好，但可增加 `computeIfAbsent` 简化逻辑
